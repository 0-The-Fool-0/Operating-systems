# Лабораторная работа №4  
## Тема: Синхронизация процессов

Проект реализует обмен сообщениями между процессами `Sender` и `Receiver` через общий бинарный файл.  
Передача организована как кольцевая очередь FIFO с межпроцессной синхронизацией средствами WinAPI.

---

## 1. Коротко о теории

### Процессы и потоки
- **Процесс** в Windows — объект ядра с собственными ресурсами (адресное пространство, дескрипторы, потоки).
- **Поток** — единица выполнения, которой планировщик выделяет процессорное время.
- В этой ЛР используются **разные процессы** (`Receiver` + несколько `Sender`), поэтому нужна синхронизация именно **межпроцессная**.

### Зачем нужна синхронизация
Если несколько процессов одновременно читают/пишут один ресурс (файл), возникают гонки данных:
- потеря сообщений,
- повреждение структуры очереди (`head/tail/count`),
- чтение «наполовину записанных» данных.

Синхронизация решает две задачи:
1. **Взаимное исключение** (критическая секция на уровне процессов).
2. **Условная синхронизация** (когда можно читать/писать, а когда надо ждать).

### Объекты синхронизации, используемые в работе
- **Mutex** — защищает критическую секцию доступа к файлу (одновременно только один процесс меняет заголовок/слоты).
- **Semaphore empty** — считает свободные слоты в очереди (для производителей `Sender`).
- **Semaphore full** — считает занятые слоты (для потребителя `Receiver`).
- **Semaphore ready** — сигнал готовности `Sender` к работе.

Это классическая схема **producer-consumer**:
- `Sender` = producer,
- `Receiver` = consumer.

### FIFO и кольцевая очередь
Очередь хранится в файле и описывается полями:
- `capacity` — число слотов,
- `head` — индекс чтения,
- `tail` — индекс записи,
- `count` — текущее количество сообщений.

После записи `tail = (tail + 1) % capacity`,  
после чтения `head = (head + 1) % capacity`.

Так сохраняется порядок FIFO и переиспользуются освобожденные места (кольцо).

### Каналы (анонимные/именованные) и связь с ЛР
В теории ОС есть IPC через:
- **анонимные каналы** (`CreatePipe`) — обычно между родственными процессами,
- **именованные каналы** (`CreateNamedPipe`) — для связи произвольных процессов/по сети.

В этой ЛР выбран другой IPC-механизм: **общий файл + объекты синхронизации ядра**.  
По смыслу это та же задача обмена данными между процессами, но на базе файлового хранилища.

## 3. Как работает код

### `Receiver`
1. Вводит имя файла и число записей.
2. Создаёт/инициализирует бинарный файл очереди.
3. Создаёт объекты синхронизации (`mutex`, `empty`, `full`, `ready`).
4. Запускает `N` процессов `Sender` через `CreateProcess`.
5. Ждёт `N` сигналов готовности через `ready`.
6. В цикле по команде:
   - `read` — ждёт `full`, захватывает `mutex`, читает сообщение, освобождает `mutex`, увеличивает `empty`;
   - `quit` — завершает работу.

Если очередь пуста, `WaitForSingleObject(full, INFINITE)` блокирует `Receiver` до появления сообщения.

### `Sender`
1. Получает имя файла из аргументов командной строки.
2. Открывает файл и уже созданные объекты синхронизации (`OpenMutex`, `OpenSemaphore`).
3. Посылает сигнал готовности через `ready`.
4. В цикле:
   - `send` — ввод сообщения, ожидание `empty`, захват `mutex`, запись, освобождение `mutex`, увеличение `full`;
   - `quit` — завершение.

Если очередь заполнена, `WaitForSingleObject(empty, INFINITE)` блокирует `Sender`.

### `QueueFile`
Отвечает за низкоуровневую файловую часть:
- запись/чтение заголовка очереди;
- `PushMessage()` и `PopMessage()`;
- инициализацию слотов;
- проверку ошибок WinAPI и формирование понятного текста ошибки (`FormatWindowsError`).

## 4. Обработка ошибок и корректность

В проекте реализовано:
- проверка корректности пользовательского ввода;
- обработка исключений `try/catch` в `Receiver` и `Sender`;
- расшифровка кодов WinAPI (`GetLastError` + `FormatMessage`);
- RAII для дескрипторов (`ScopedHandle`) — исключает утечки ресурсов.

Также учтены практические требования:
- нет `using namespace std`;
- используются приведения в стиле C++ (`static_cast`, `reinterpret_cast`);
- выделение/освобождение ресурсов организовано корректно.

---

## 5. Unit-тесты

Тесты в `tests/QueueFileTests.cpp` проверяют:
1. **FIFO + wrap-around** (корректная работа кольцевой очереди).
2. **Overflow** (ошибка при записи в полную очередь).
3. **Underflow** (ошибка при чтении из пустой очереди).

Запуск через CTest:
```bash
ctest -C Release --test-dir build --output-on-failure
```

## 6. Сборка и запуск

### Сборка (CMake)
```bash
cmake -S . -B build
cmake --build build --config Release
```

### Запуск
1. Запустить `Receiver`:
```bash
build/Release/Receiver.exe
```
2. Ввести:
   - имя бинарного файла (например, `messages.bin`);
   - размер очереди (число записей);
   - число процессов `Sender`.
3. В окнах `Sender` использовать команды:
   - `send` — отправить сообщение;
   - `quit` — выйти.
4. В `Receiver`:
   - `read` — читать следующее сообщение FIFO;
   - `quit` — завершить работу.

## 7. Что важно для защиты

Нужно уметь объяснить:
- почему для межпроцессного доступа нужен именно **именованный mutex/semaphore**;
- как `empty/full` предотвращают переполнение и чтение из пустой очереди;
- почему без `mutex` даже при семафорах возможна порча `head/tail/count`;
- как реализован FIFO в бинарном файле;
- чем этот подход отличается от IPC через pipe (анонимный/именованный канал).
